# 接口分层与寄存器设计

## 一、设计原则

对外接口要简单，对内控制可以保留复杂度。

所以推荐分两层：

- **应用层接口**
  - 给 CPU/C 固件用
  - 表达“任务”
- **底层控制接口**
  - 给 `frontend` 和 debug 用
  - 表达“微步骤”

正式软件只看应用层接口。

## 二、推荐接口分层

```text
CPU software
  -> 应用层寄存器
  -> TPU App Wrapper
  -> frontend 内部接口
  -> TPU core

debug/testbench
  -> debug page 或仿真专用口
  -> IMEM / UB / step
```

## 三、第一阶段应用层寄存器

建议地址映射：

| 地址偏移 | 名称 | 说明 |
|---|---|---|
| `0x00` | `CTRL` | bit0=`start`，bit1=`soft_reset`，bit2=`irq_en` |
| `0x04` | `STATUS` | bit0=`busy`，bit1=`done`，bit2=`error` |
| `0x08` | `MODE` | bit0=`infer/train`，bit1=`use_desc` |
| `0x0C` | `NET_ID` | 选择网络配置 |
| `0x10` | `IN0` | 输入特征 0，Q8.8 |
| `0x14` | `IN1` | 输入特征 1，Q8.8 |
| `0x18` | `OUT0` | 输出 0，Q8.8 |
| `0x1C` | `OUT1` | 输出 1 或类别 |
| `0x20` | `LR` | 学习率，训练模式使用 |
| `0x24` | `AUX0` | 扩展参数位 |
| `0x28` | `DESC_LO` | 第二阶段 descriptor 地址低位 |
| `0x2C` | `DESC_HI` | 第二阶段 descriptor 地址高位/保留 |
| `0x30` | `IRQ_STATUS` | 中断状态，写 1 清 |
| `0x34` | `DBG_CYCLE` | 调试运行周期计数 |

第一阶段最小闭环实际只需要：

- `CTRL`
- `STATUS`
- `MODE`
- `NET_ID`
- `IN0`
- `IN1`
- `OUT0`
- `OUT1`

## 四、底层 debug 页

底层寄存器不建议继续作为正式应用接口，但建议保留成 debug 页：

| 地址偏移 | 名称 | 说明 |
|---|---|---|
| `0x100` | `IMEM_ADDR` | debug: 选择 IMEM 槽位 |
| `0x104` | `IMEM_W0` | debug: 写入微指令 |
| `0x108` | `IMEM_WE` | debug: 提交写 IMEM |
| `0x10C` | `IMEM_LEN` | debug: 有效指令长度 |
| `0x110` | `UB_DATA0` | debug: lane0 数据 |
| `0x114` | `UB_DATA1` | debug: lane1 数据 |
| `0x118` | `UB_PUSH` | debug: 推送 UB |
| `0x11C` | `STEP` | debug: 单步发射 |

这页的用途：

- bring-up
- 波形定位
- 验证隔离
- 微程序调试

它的定位是：

- **保留**
- **不让应用层长期依赖**

## 五、为什么底层寄存器对验证好

因为它们能直接控制微结构，帮助你把问题拆层定位。

例如 TPU 输出错误时，可以分层验证：

- 高层 API 是否写对
- wrapper 是否锁存对
- frontend 是否发对 `IMEM`
- core 是否执行对

有底层 debug 页时，你可以直接：

- 写 `IMEM`
- 推 `UB`
- 单步 `step`
- 看 `wait_after`
- 看 `vpu_drain`

这样能快速判断是：

- 软件问题
- wrapper 问题
- sequencer 问题
- 算子核问题

## 六、App Wrapper 和 frontend 的内部接口

建议不要让 wrapper 直接长期操作一堆微寄存器，而是用更干净的内部信号：

- `launch`
- `mode`
- `net_id`
- `desc_base`
- `busy`
- `done`
- `error`

也就是：

- wrapper 负责“这次跑什么任务”
- frontend 负责“把任务拆成微步骤并安全执行”

## 七、descriptor 的第二阶段设计

第二阶段开始支持 descriptor。

建议先做一个 MLP/小网络专用 descriptor，而不是一开始追求全通用。

```c
typedef struct {
    uint32_t flags;      // bit0=infer, bit1=train, bit2=irq_en
    uint32_t net_id;
    uint32_t x_addr;
    uint32_t y_addr;
    uint32_t out_addr;
    uint32_t w1_addr;
    uint32_t b1_addr;
    uint32_t w2_addr;
    uint32_t b2_addr;
    uint16_t input_dim;
    uint16_t hidden_dim;
    uint16_t output_dim;
    uint16_t lr_q8_8;
    uint16_t batch;
    uint32_t next_desc;
} tpu_desc_t;
```

descriptor 的价值：

- 参数多时不必写很多寄存器
- 容易扩网络结构
- 容易接 DMA
- 容易接 custom instruction

## 八、为什么现在先上 AXI-Lite

当前阶段推荐：

- **控制面** 用 `AXI-Lite`
- **数据面** 以后再扩 `DMA`

原因：

- `AXI-Lite` 足够承载控制寄存器
- 容易集成到 CPU SoC
- 便于答辩时讲清楚 SoC 化
- 不需要一步到位做复杂内存搬运系统

## 九、DMA 的后续演进

后续数据面演进路线：

1. 第一阶段：
   - CPU 直接通过寄存器或片上 RAM 送数据
2. 第二阶段：
   - descriptor 指定地址
3. 第三阶段：
   - TPU 作为 `AXI master`
   - DMA 从 SRAM/DDR 拉输入和权重
   - DMA 回写结果

这条路和当前 `AXI-Lite` 并不冲突。

## 十、自定义指令和当前接口的关系

当前可以先做：

- C driver API
- `AXI-Lite` 寄存器启动

后续再做：

- 一条 `custom instruction`
- 这条指令的本质只是“更优雅地启动任务”

所以：

- `custom instruction` 是**启动方式升级**
- 不是替代整个 `AXI-Lite + wrapper + frontend` 架构

## 十一、AXI Arbiter 在后续系统里的位置

当前若只有：

- `CPU` 一个 master
- `TPU/UART/RAM` 几个 slave

则仲裁器不是第一优先级。

后续当系统扩展为：

```text
CPU master -----------\
                        -> AXI arbiter/interconnect -> SRAM/DDR
TPU DMA master -------/
```

这时仲裁器就成了系统关键基础设施：

- 解决 CPU 与 TPU DMA 抢内存
- 体现你对大 SoC 的系统理解

所以 AXI arbiter 非常有用，但它更偏第二阶段到第三阶段的加分项。
