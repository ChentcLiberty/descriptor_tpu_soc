# 嵌入式软件与演示路径

## 一、怎么在当前基础上做嵌入式

这里的“嵌入式”不是另起炉灶，而是：

- 把 TPU 做成一个 SoC 外设
- 让 CPU 跑固件去控制它
- 完成一个具体应用任务：呼吸识别

所以嵌入式部分实际上包含三层：

- 硬件系统
- 固件驱动
- 应用逻辑

## 二、推荐软件目录

建议新增一个简单的软件目录：

```text
sw/
  startup.S
  main.c
  uart.c
  uart.h
  tpu_driver.c
  tpu_driver.h
  breath_app.c
  breath_app.h
```

职责建议：

- `startup.S`
  - 启动和栈初始化
- `main.c`
  - 系统初始化和 demo 调度
- `uart.c`
  - 串口打印
- `tpu_driver.c`
  - TPU 寄存器访问、启动、等待、读结果
- `breath_app.c`
  - 呼吸信号预处理、特征提取、分类封装

## 三、推荐 C API

第一阶段不要先上 custom instruction，先把 C driver API 做稳。

建议接口：

```c
void tpu_write_input(int16_t x0, int16_t x1);
void tpu_start(uint32_t mode, uint32_t net_id);
void tpu_wait_done(void);
void tpu_read_output(int16_t *y0, int16_t *y1);
int  breath_classify(int16_t feat0, int16_t feat1);
```

上层应用只关心：

```c
cls = breath_classify(feat0, feat1);
```

## 四、custom instruction 在当前项目里的定位

`custom instruction` 是后续升级项，不是第一阶段的必须项。

它的作用是：

- 把“启动 TPU”这件事，从写寄存器升级成一条 CPU 指令

但即使以后加了 custom instruction，通常也只是启动方式升级，不会替代：

- wrapper
- frontend
- descriptor
- DMA

## 五、第一阶段 demo 运行流程

```text
开机
-> 初始化 UART / TPU driver
-> 读取或内置一段呼吸样本
-> CPU 预处理并提取 2 个特征
-> CPU 写 TPU 的 IN0/IN1/MODE
-> CPU 写 START
-> TPU 运行
-> CPU 轮询 DONE
-> CPU 读 OUT/CLASS
-> UART 打印分类结果
```

## 六、为什么先把呼吸信号压成 2 个特征

因为当前 TPU 只有很小的 `2x2 MLP`，更适合做：

- 小规模分类器

不适合第一阶段直接吃原始长时序。

所以当前推荐：

- CPU 负责时域/频域特征提取
- TPU 负责最后的神经网络分类

这是最现实的工程拆分。

## 七、训练和推理怎么在嵌入式里体现

第一阶段建议：

- 固件里保留 `MODE=infer/train`
- 先把 `infer` 路径跑通

如果时间允许，再做：

- 小规模训练路径

固件接口可以统一：

```c
void tpu_start(uint32_t mode, uint32_t net_id);
```

这样以后扩训练，不会改掉整个软件框架。

## 八、嵌入式演示时最重要的东西

你要让面试官或评委看到的不是“某个波形”，而是：

- CPU 在跑固件
- CPU 能控制 TPU
- TPU 能完成推理
- 系统能输出识别结果

也就是：

**这是一个能完成具体任务的嵌入式智能系统，而不是单个 RTL 核。**

## 九、建议的阶段化演示内容

### 演示 1：最小闭环

- 手工写两个特征
- TPU 输出分类

### 演示 2：系统闭环

- 一段呼吸样本
- CPU 提取特征
- TPU 分类
- UART 打印结果

### 演示 3：加分项

- 切换 `MODE`
- 切换 `NET_ID`
- 打印运行周期

## 十、后续嵌入式扩展路径

嵌入式这条线后续很自然可以继续扩：

- descriptor
- DMA
- custom instruction
- 中断
- 更复杂网络
- 实时采样和在线识别

所以当前做法不是临时 demo，而是后续 SoC 化和产品化的基础。
