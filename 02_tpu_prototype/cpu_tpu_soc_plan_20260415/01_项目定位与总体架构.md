# 项目定位与总体架构

## 一、项目定位

这个项目的正确定位不是“通用 TPU 编译器”，而是：

**一个面向呼吸识别任务的 CPU+TPU 异构 SoC 原型。**

当前阶段目标：

- 把现有 TinyTPU 计算核接到处理器系统里
- 通过 `AXI-Lite` 实现控制
- 先跑通固定小规模 MLP inference
- 把项目讲成一个完整的异构嵌入式系统

后续阶段再扩：

- 训练
- descriptor
- DMA
- custom instruction
- 1D CNN

## 二、这个 TPU 里已经有哪些算子

当前项目不是“没有算子”，而是“算子已经被串成固定数据流”。

已有核心算子：

- `matmul / outer-product`
  - 由脉动阵列完成
  - 代码在 `src/systolic.sv`
- `bias`
- `leaky_relu`
- `loss_derivative`
- `activation_derivative`
  - 这些由 `VPU` 组合完成
  - 代码在 `src/vpu.sv`
- `gradient update`
  - 代码在 `src/gradient_descent.sv`

所以系统层可以表达为：

- CPU 负责不规则任务
- TPU 负责规则、重复、并行度高的神经网络算子

## 三、推荐系统架构

```text
                +----------------------+
                |      CPU Core        |
                |  C firmware / app    |
                +----------+-----------+
                           |
                           | AXI-Lite / MMIO
                           v
                +----------------------+
                |   TPU App Wrapper    |
                |  app regs / status   |
                |  mode / net select   |
                +----------+-----------+
                           |
                           | internal launch/config
                           v
                +----------------------+
                |      frontend        |
                | IMEM + sequencer     |
                | wait + CU            |
                +----------+-----------+
                           |
                           v
                +----------------------+
                |     TPU Compute      |
                | systolic + VPU + UB  |
                | update               |
                +----------------------+
```

## 四、三层职责划分

### 1. CPU

负责：

- 呼吸信号预处理
- 特征提取
- 模式控制
- 启动 TPU
- 等待完成
- 读回结果
- 串口/显示输出

### 2. TPU App Wrapper

这是新增的“高层壳子”，面向 CPU。

负责：

- 暴露简单寄存器接口
- 锁存参数
- 选择 `infer/train`
- 选择 `NET_ID`
- 启动 frontend
- 汇总 `busy/done/error`
- 后续接 descriptor / IRQ / DMA

### 3. frontend

现有 `frontend` 不是最终应用接口，而是加速器内部控制器。

负责：

- `IMEM`
- `sequencer`
- `wait_after / vpu_drain`
- `control_unit`
- 对 TPU core 发底层控制信号

### 4. TPU core

负责真正计算：

- systolic
- VPU
- UB
- gradient update

## 五、为什么不建议删 frontend

不删的原因不是“舍不得旧代码”，而是它已经包含了最难的控制资产：

- 微指令格式
- 阶段调度
- `wait_after`
- `vpu_drain`
- core 低层控制出口

这些后续做 descriptor、DMA、训练和更复杂网络时仍然有价值。

正确做法不是删，而是：

- **保留 frontend 的控制内核**
- **把它从 CPU 直接使用的 AXI 外设，降级为内部微控制层**

## 六、为什么不让 CPU 直接长期操作底层微寄存器

如果 CPU 直接操作：

- `IMEM_W0`
- `IMEM_WE`
- `UB_DATA`
- `UB_PUSH`
- `step`

那 CPU 软件等于在“手工操纵微架构”，而不是“调用一个加速器任务”。

这对验证很好，但对大 SoC 和应用开发不好。

## 七、这个分层有没有做大的风险

有风险，但风险可控，而且比“CPU 直接碰微结构接口”更小。

主要风险：

- `App Wrapper` 和 `frontend` 职责打架
- wrapper 知道太多底层细节
- debug 接口和正式接口混在一起

规避原则：

- `App Wrapper` 只做任务级控制
- `frontend` 只做微程序级控制
- 正式接口和 debug 接口分开

## 八、为什么不是所有任务都交给 TPU

不是所有任务都适合加速器。

CPU 更适合：

- 控制流
- 预处理
- 特征提取中的不规则逻辑
- 串口/显示/系统管理

TPU 更适合：

- 矩阵乘
- 卷积
- 激活
- update
- 规则、重复、并行的神经网络核心计算

## 九、呼吸识别任务怎么落到这套系统上

推荐第一阶段 demo：

- 输入一段呼吸样本
- CPU 提取 2 个特征
- TPU 跑 `2x2` 小 MLP
- CPU 读回分类结果

这条路径最符合当前硬件能力，也最容易做成完整演示。
