# TPU 项目高频面试题与标准答法

这份文档只解决一件事：
**把面试里最可能被问到的问题，压成你能直接开口说的答案。**

建议使用方式：
1. 先把前 8 题背顺。
2. 再把 `UB / compiler / verification / 个人贡献` 这四块加深。
3. 每道题先记住“短答版”，再记“展开版”。

---

## 1. 这个项目到底做成了什么？

### 短答版
我把 tiny-tpu 从一个偏裸核的原型，补成了一个可以由 Host 通过 AXI-Lite 控制、可以跑完整训练闭环、并且能用波形和收敛测试证明正确性的 SoC 级原型。

### 展开版
原来的 tiny-tpu 更像几个计算模块的集合，缺少真正的系统级控制面、软件到硬件的控制链、训练闭环的数据回写路径，以及可验证的 SoC 顶层。我这次补了四层东西：
- 顶层 SoC wrapper，让 Host 可以通过 AXI-Lite 控制 TPU。
- Frontend + IMEM + sequencer，把软件生成的控制字按节奏发给硬件。
- Unified Buffer、PE、VPU 组成的训练数据闭环，也就是 `UB -> systolic -> VPU -> UB`。
- 端到端验证，包括波形、回归和 train convergence。

### 一句收口
所以这个项目不是单点 demo，而是一条从软件、控制、执行到验证都闭合的系统链。

---

## 2. 为什么你说它不是推理 demo，而是训练闭环？

### 短答版
因为数据不是从阵列出来就结束了，VPU 的输出会重新写回 UB，后面还能继续参与 backward 和 parameter update。

### 展开版
如果只是推理 demo，通常做到 `UB -> PE -> 输出` 就结束了。但这里不是。`tpu.sv` 里很清楚地把 `vpu_data_out_* / vpu_valid_out_*` 接回了 UB 写口，所以 forward 产生的中间结果、backward 的梯度、以及更新后的参数，都能回到 UB 再被后续阶段消费。再结合 VPU 的 pathway 设计，比如 `1100` 表示 forward、`1111` 表示 transition/loss、`0001` 表示 backward derivative，就能形成多阶段训练流程，而不是一次性推理流水。

### 一句收口
判断它是不是训练闭环，关键不是有没有 PE，而是结果有没有回写并继续参与下一阶段。

---

## 3. Host 是怎么控制 TPU 的？

### 短答版
Host 通过 AXI-Lite 配置 Frontend 寄存器和 IMEM，Frontend 再按 sequencer 节奏把指令 decode 成 UB、systolic、VPU 的控制字段。

### 展开版
整个控制链是：
- Host 先通过 AXI-Lite 把程序写进 IMEM，把数据写进 UB。
- 然后写 `CTRL.start` 启动 Frontend。
- Frontend 内部的 sequencer 按顺序取 IMEM 指令。
- `control_unit.sv` 把 32-bit 指令拆成 `ub_rd_start / ub_ptr_sel / row / col / transpose / vpu_pathway / sys_switch` 这些字段。
- 这些字段最终进入 `tpu.sv`，驱动 UB 发数、PE 阵列切权重、VPU 切路径。

### 一句收口
Host 不直接一拍一拍控 PE，而是通过 Frontend 把“程序”翻译成硬件节奏。

## 4. 编译器是怎么和 RTL 对上的？

### 短答版
编译器的 `scheduler.py` 负责生成 stage-level 的控制序列，`control_unit.sv` 负责把这些 32-bit 控制字 decode 成 RTL 真正消费的字段。

### 展开版
软件侧最关键的是 `_ub_read()`、`_switch()`、`_wait()`、`_nop()` 这些 helper。它们生成的不是抽象概念，而是非常具体的字段，比如：
- `ub_ptr_select`
- `ub_rd_addr_in`
- `ub_rd_row_size`
- `ub_rd_col_size`
- `ub_rd_transpose`
- `vpu_data_pathway`
这些字段最终被编码进 IMEM。到了 RTL 侧，`control_unit.sv` 按位切出 `opcode / addr / row / col / transpose / ptr / pathway`，直接驱动后级模块。所以软件和硬件不是 loosely coupled，而是 bit-level 对齐。

### 一句收口
这个项目的软件硬件接口，不是靠文档约定，而是靠指令位段一一对应。

---

## 5. Frontend 最核心的价值是什么？

### 短答版
Frontend 的价值不是“有寄存器”，而是把 Host 的配置、IMEM 的程序和执行阶段的节奏真正闭合起来。

### 展开版
Frontend 里最重要的是三件事：
- IMEM：把控制程序存起来，而不是靠 Host 一拍一拍推命令。
- Sequencer：负责 `idle -> run -> wait -> drain -> done` 这类阶段切换。
- Status / handshake：比如 `wait_after -> vpu_drain`，要根据 core 的 valid 回来判断什么时候这一阶段真的结束。
此外还有一个很工程化但很关键的点，就是 Host 写 UB 的入口和 control_unit 触发的 host write 要做仲裁，不然系统里会出现写口冲突。

### 一句收口
Frontend 真正解决的是“软件程序如何变成硬件可执行时序”。

## 6. Unified Buffer 为什么是最难的模块？

### 短答版
因为它不是一个普通 RAM，而是训练过程中所有数据流、指针语义和阶段切换的汇合点。

### 展开版
UB 同时要处理很多事情：
- Host 初始装载。
- Input 和 weight 的不同读流语义。
- `ptr_select` 决定当前到底读的是 input、weight、bias、Y、H 还是旧参数。
- `wr_ptr_base / restore` 保证不同阶段写回和恢复的边界是对的。
- VPU 写回的数据后面还要继续被读出来参与下一阶段。
- 甚至还有部分 parameter update 逻辑在 UB 内完成。
所以 UB 难不是因为代码最长，而是因为它是系统语义最集中的地方，一旦指针和时序错一个拍，后面整条链都会错。

### 一句收口
UB 难在“语义密度”和“阶段边界”，不是难在存储体本身。

---

## 7. `wr_ptr / base / restore` 为什么值得单独讲？

### 短答版
因为训练是多阶段的，写指针如果不在阶段边界恢复，后续写回就会覆盖错区域，整个闭环都会乱掉。

### 展开版
训练不是一段线性流，而是 forward、transition、backward、update 多阶段串起来的。每一阶段 VPU 写回的数据语义不同，如果写指针只是单纯递增，不做 `base` 和 `restore`，那下一阶段可能会把数据写到不该写的区间，或者读的时候读错上一阶段的结果。`wr_ptr_base` 负责记录阶段基准，`restore` 负责在正确边界回到该回的位置。这件事一旦错了，波形上可能只是晚几拍暴露，但最后会直接表现成 loss 不收敛。

### 一句收口
`restore` 不是优化项，它是训练多阶段能不能闭合的基础条件。

## 8. PE 阵列和 VPU 在系统里分别负责什么？

### 短答版
PE 阵列负责高吞吐乘加，VPU 负责训练路径相关的逐元素运算和路径切换。

### 展开版
PE 阵列，也就是 systolic array，核心职责是做矩阵乘加和波前传播。它吃的是：
- 左边界输入流
- 上边界权重流
- `sys_switch` 这样的阵列控制信号
VPU 则负责训练里那些不适合直接塞进阵列的数据路径，比如 bias、激活、导数、loss 相关运算，以及不同训练阶段的 pathway 切换。所以这两个模块的分工是：
- PE 提供高效 MAC 计算
- VPU 提供训练语义和逐元素处理

### 一句收口
PE 决定算得快不快，VPU 决定训练语义是不是完整。

---

## 9. `sys_switch` 的意义是什么？

### 短答版
它是 shadow weight 切到 active weight 的边界信号，保证阵列不会在装权重过程中就提前参与计算。

### 展开版
训练或推理前，权重往往先被装到 shadow 区。只有当当前 tile 或阶段需要真正开始计算时，才通过 `SWITCH` 指令打一拍 `sys_switch_in`，告诉阵列把 shadow 内容切到 active。这样做的好处是，装载和执行解耦，不会边装边算造成时序和数据混乱。编译器里 `_switch()` 的意义，最终在 RTL 上就落成这个单拍信号。

### 一句收口
`switch` 是权重“生效”的边界，不是单纯的状态位。

## 10. 你怎么证明系统真的跑对了？

### 短答版
我不是只看单点波形，而是用三层证据：波形、回归、训练收敛。

### 展开版
第一层是波形证据，看控制信号、UB 发数、PE 输出、VPU valid 这些关键路径是不是按预期推进。第二层是回归和端到端测试，确认 AXI 装载、IMEM 执行、状态寄存器、数据读回都正确。第三层是训练收敛，也就是 `test_tpu_soc_axil_train_convergence.py` 这种真正能看到 loss 下降或参数朝正确方向变化的系统级证据。只有这三层叠起来，才能说明它不是“看起来有波形”，而是真的系统正确。

### 一句收口
波形能证明时序，收敛能证明语义，二者缺一不可。

## 11. 你做的关键 bug 修复有哪些？

### 短答版
我做的修复集中在阶段边界和控制闭环上，比如 `wait_after`、`pathway hold`、`host/CU write mux`、`wr_ptr restore` 这些点。

### 展开版
这类 bug 的共同特点是：
- 单看模块可能不明显。
- 一旦到系统级就会放大成错误的阶段切换、错误写回或不收敛。
举几个典型例子：
- `wait_after`：如果没有正确等到 `vpu_drain`，Frontend 会过早推进到下一阶段。
- `pathway hold`：如果 VPU pathway 过早切掉，当前阶段结果语义会错。
- `host/CU mux`：两路 host write 不仲裁好会直接导致 UB 写入口混乱。
- `wr_ptr restore`：恢复边界不对，后续读写区域会串。

### 一句收口
我修的不是表面 bug，而是系统闭环里最容易让训练语义失真的边界 bug。

## 12. 这个项目里你个人最有含金量的贡献是什么？

### 短答版
我认为最有含金量的不是某个单模块，而是把软件、控制、执行和验证真正接成闭环，并把几个最容易在阶段边界出错的点修稳。

### 展开版
如果只写一个模块，项目还只是局部可运行；真正有价值的是把：
- 编译器控制序列
- Frontend sequencer
- UB / PE / VPU 数据闭环
- SoC 顶层接入
- Train convergence 验证
这几层串起来。这里面最难的不是写大块代码，而是保证阶段边界、路径语义、写回区域、完成条件全部对齐。也就是说，贡献的核心是系统工程能力，而不是某个 isolated RTL 技巧。

### 一句收口
我最核心的贡献，是把这个项目从“模块能跑”推进到“系统闭环可证明”。

## 13. 如果继续做，下一步会做什么？

### 短答版
我会先补 DMA/中断这类系统能力，再做更大规模阵列和更完整的软件栈。

### 展开版
当前系统已经是可控、可执行、可验证的训练原型，但工程上还有几个自然下一步：
- 把 AXI-Lite 扩到 DMA 驱动，减少 Host 搬运开销。
- 补 IRQ，让 Frontend/TPU 的完成事件可以异步通知 Host。
- 扩阵列规模，比如更大 tile 或更宽数据通路。
- 继续完善编译器，把 schedule 自动化做得更完整。
- 继续补验证矩阵，覆盖更多模型和边界场景。

### 一句收口
下一步的重点不是推翻架构，而是在现有闭环上把系统能力做完整。

## 14. 为什么当前编译器不是 cycle-accurate，`vpu_drain` 会不会影响性能？

### 短答版
因为当前目标是先把系统闭环跑通，所以编译器先做阶段级调度，而不是逐拍最优调度。`vpu_drain` 会影响性能，因为它是保守的等待边界，但它先保证了阶段语义不串。

### 展开版
当前 `scheduler.py` 更像一个阶段级微程序生成器。它已经能生成：
- UB 布局
- host 装载计划
- stage-level schedule
- IMEM 指令

但它还不能精确建模：
- UB、PE、VPU 各条路径的真实逐拍延迟
- 同一拍资源会不会冲突
- 哪些等待可以压缩到最少

所以现在很多地方还是固定 `nop(4)` 或规则式 `wait_after`，优先保证系统先跑通。

`vpu_drain` 的作用也是一样。当前 Frontend 会在某些命令后等到 VPU valid 全部掉下去，再进入下一阶段。这样做很安全，因为它保证前一阶段尾拍真的吐干净了，后一阶段才开始；但代价是它偏保守，会牺牲吞吐。

真正更好的改进方向不是把 `vpu_drain` 直接删掉，而是：
- 做更精确的 latency model
- 做资源冲突建模
- 自动插入最少等待
- 把全局 `vpu_drain` 换成更细粒度的 done 信号

### 一句收口
当前版本的关键词是“先闭环、后榨性能”；`vpu_drain` 先保证正确性，后续再用更细的模型把等待压短。

## 15. 如果面试官让你只讲一个最难点，你讲什么？


### 短答版
我会讲 UB 和阶段边界，尤其是 `ptr_select`、`wr_ptr_base/restore` 和 UB-PE 时序对齐。

### 展开版
因为这块最能体现这个项目不是拼模块，而是真正理解训练数据流。表面上看 UB 像内存，实际上它承载了：
- 多种语义数据区
- 多阶段写回
- 指针恢复
- 输入和权重的不同发数时序
- 和 PE/VPU 的对齐
这块讲清楚，基本就能说明我不是只会复述模块名，而是理解训练闭环为什么能成立。

### 一句收口
如果只能讲一个难点，我讲 UB，因为它最能代表系统理解深度。

## 16. 面试最后 30 秒怎么收尾？

### 短答版
我会说，这个项目的价值不在于做了一个小 TPU，而在于把软件、控制、执行和验证真正接成了可证明的训练闭环。

### 展开版
我的收尾通常会落在三点：
- 第一，它已经不是单点计算 demo，而是 SoC 级原型。
- 第二，它能从编译器控制序列一路落到 RTL 执行字段。
- 第三，它不是只看波形，而是能用 train convergence 证明系统正确。
所以这套项目最重要的能力，不是单模块设计，而是系统闭环和跨层联调。

### 一句收口
这就是我认为这个项目最能体现工程价值的地方。
