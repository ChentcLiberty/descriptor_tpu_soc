# TinyTPU AXI-Lite SoC 面试高频追问清单

## 1. 这个项目你到底做了什么？

最稳回答：
- 我做了三块：SoC 前端与控制链路、IMEM 指令化执行、cocotb 闭环验证。
- 我不是只写了一个接口壳，而是把原来主要靠 testbench 直驱的 tiny-tpu，补成了可控、可验证的 SoC 原型。

## 2. 为什么要加 AXI-Lite 前端？

最稳回答：
- 原始 tiny-tpu 更偏裸 RTL，很多控制依赖 testbench 直接驱动。
- 加 AXI-Lite 前端之后，外部可以通过寄存器配置参数、写 UB、写 IMEM、启动执行，这样才更像真实 SoC 原型。

## 3. Sequencer 的价值是什么？

最稳回答：
- Sequencer 把执行流程从“testbench 手搓时序”变成“IMEM 指令驱动”。
- 这样可以把一整套 forward、backward、update 流程编码进 IMEM，执行路径更稳定，也更容易复现和验证。

## 4. 你这个“编译器”是不是通用编译器？

最稳回答：
- 不是。
- 我会把边界说清楚：这是当前 tiny-tpu 原型的 stage-level scheduler 和 IMEM 编码链路，目标是驱动当前 2-layer MLP 配置，而不是通用编译器。

## 5. 为什么 `vpu_data_pathway` 要保持？

最稳回答：
- 因为 systolic 的输出不是 dispatch 当拍就到，而是会晚几拍。
- 如果 pathway 只在当前拍有效，后面几拍 VPU 就会错误 bypass，所以必须在 dispatch 时 latch 住，覆盖后续流水线延迟。

## 6. `wait_after` 解决了什么问题？

最稳回答：
- 早期等待语义靠隐式规则推断，会错误消费 drain 事件，导致 sequencer 卡死。
- 改成 IMEM 显式 `wait_after` 之后，等待语义清楚，状态机更稳定，也更容易 debug。

## 7. 你怎么证明训练是可信的？

最稳回答：
- 我做了两层验证。
- 第一层是单次 e2e，逐项检查 H1、dZ2、dZ1 和 UB update，结果是 `41/41 PASS`。
- 第二层是多 epoch 收敛，看系统级行为是否正确，结果是 12 个 epoch 后 loss 从 `0.2529` 降到 `0.1777`，最终 XOR 分类正确。

## 8. 为什么不是每一拍都和 NumPy 完全一致？

最稳回答：
- 因为这里是 fixed-point Q8.8，实现里有截断、量化和流水线时序差异。
- 所以系统级验证更看重行为正确、趋势正确和最终任务正确，而不是强行要求所有中间值逐拍完全一致。

## 9. 你觉得项目里最有代表性的 bug 是哪个？

最稳回答：
- 我一般讲三个：host write 无效、`vpu_data_pathway` 不保持、wait 逻辑卡死。
- 这三个分别对应接口采样、流水线控制保持和状态机等待语义，能比较完整体现工程能力。

## 10. 现在这个 SoC 能不能直接说是完整 SoC？

最稳回答：
- 不能这么说。
- 更准确的说法是：当前已经补齐了 AXI-Lite 控制、IMEM、sequencer 和验证闭环，但 DMA、IRQ、软件 runtime 等完整 SoC 配套还不在当前范围内。

## 11. 项目最硬的结果是什么？

最稳回答：
- `41/41 PASS`
- `12 epoch loss 0.2529 -> 0.1777`
- `XOR = (0,1,1,0)`
- pipeline 变体 `164.10 -> 183.91 MHz`

## 12. 如果面试官追问“你最想体现的能力是什么”？

最稳回答：
- 我最想体现的是系统集成能力和验证闭环能力。
- 不是只会写模块，而是能把接口、控制链路、执行链路和验证闭环一起打通，并且把 bug 收敛到信号级。

## 13. 如果面试官问“还有什么没做完”？

最稳回答：
- 我会直接说边界：当前没有扩到 DMA、IRQ 和更通用的软件 runtime。
- 另外 scheduler 也没有扩成通用编译器，这部分我不会夸大。

## 14. 最后 20 秒总结怎么说？

最稳回答：
- 我把 tiny-tpu 从主要靠 testbench 直驱的裸 RTL，补成了带 AXI-Lite 前端、IMEM 和 sequencer 的 SoC 原型。
- 我又用 cocotb 把单次 e2e 和多 epoch 训练闭环跑通，所以这个项目最能体现的是系统集成、控制链路和验证闭环能力。
