# 高频问答口径

> 更新时间：2026-04-02
> 原则：先结论，再证据，再边界

## Q1：你这个项目里最核心的工作是什么？

“核心不是只写了一个接口壳，而是把 tiny-tpu 从裸 RTL 形态补成了 AXI-Lite SoC 原型，并把单次 e2e 和多 epoch 训练闭环验证做通。证据上，接口和控制在 `src_axi/tpu_frontend_axil.sv`、`src_axi/tpu_soc.sv`，单次 e2e 在 `test/test_tpu_soc_axil_e2e.py`，2026-04-02 复跑是 41/41 PASS；训练收敛在 `test/test_tpu_soc_axil_train_convergence.py`，12 个 epoch 后 loss 从 0.2529 降到 0.1777。边界是当前规模固定在 Q8.8、2x2 array、2-layer MLP。”

## Q2：你怎么证明不是只把 AXI 接口接上了？

“如果只是接接口，我不会去验证 backward 和参数更新。我的 e2e 用例除了 H1，还会检查 dZ2、dZ1 和 UB 里更新后的 W1/B1/W2/B2，所以它验证的是整个执行链路，不只是寄存器可访问。这个用例当前是 41/41 PASS。”

## Q3：你项目里最难定位的 bug 是什么？

“我会举 sequencer wait 逻辑那个。现象是流程会在某个 pc 永久卡住，表面上像状态机问题，但再往下看其实是不同阶段共用了一个 `drain` 事件，前一条指令把它消费掉，后一条还在等。我最后把等待条件改成 IMEM 里的显式 `wait_after` 位，只给真正需要等待的指令打标，状态机就稳定了。”

## Q4：你这个训练结果为什么可信？

“我把训练验证拆成两层。单次 e2e 先用 numpy 参考模型做强对齐，检查中间量和更新后参数，保证一次闭环是对的；多 epoch 用例再看系统层 loss 和最终分类是否收敛。这样比只看最后 loss 更可信，也比要求多轮每个参数逐 bit 对齐更符合 fixed-point 真实情况。”

## Q5：多 epoch 为什么不要求每轮参数都和参考模型完全一致？

“因为 fixed-point 截断和 dataflow 次序差异会在多轮训练里逐步累积，小偏差是预期现象。对这个层级的验证，我更关心 loss 是否持续下降、最终分类是否正确、参数更新方向是否合理。单轮严格比对和多轮行为验证，我是分层做的。”

## Q6：这个项目的边界是什么？

“我会明确说边界：当前验证最扎实的是 Q8.8、2x2 array、2-lane、2-layer MLP/XOR 这条链路；scheduler 是当前原型的 stage-level 版本；AXI 是 Lite 控制面，不是通用高吞吐数据面。我不会把这套原型讲成一个已经泛化完成的大规模加速器。”

## Q7：如果让你继续往下做，你下一步会做什么？

“我会优先做三件事。第一，把 scheduler 从当前固定规模推广到更一般的 tile 映射。第二，把验证从 XOR/2-layer MLP 扩到更大一些的用例，并补更多 corner case。第三，如果目标偏实现，可以继续做 CDC/ECC 和 timing 收敛，这些在 `docs/CDC_IMPLEMENTATION_PLAN.md`、`docs/ECC_IMPLEMENTATION_PLAN.md`、`docs/SYN_CODE_CHANGES.md` 里都有延伸线索。”

## Q8：你做过综合或时序优化吗？

“做过一个独立 pipeline 变体。思路是在 VPU 到 UB 的写回路径之间插一级寄存器，切断关键路径。`docs/SYN_CODE_CHANGES.md` 里记录的结果是 Fmax 从 164.10MHz 提到 183.91MHz，代价是额外 1 个 cycle latency。这里我会明确说这是实验分支的结果，不是默认 AXI 路径的现成量产结论。”

## Q9：如果问到你没做深的点，怎么收口？

“这个点我知道基本原理，但我自己真正做深的是当前 tiny-tpu 原型的接口、验证和 debug 收敛；如果再往更大规模映射或者更通用编译器展开，我可以讲思路，但不会说成自己已经做过完整实现。”

## Q10：你希望面试官最后怎么评价你？

“我希望对方觉得我是那种基础不飘、项目真实、能把问题收敛到模块和信号级，而且边界很清楚的人。对实习岗来说，我觉得这比把单点包装得很强更重要。”
