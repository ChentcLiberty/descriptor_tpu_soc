# 海思风格表达与模拟面试清单

这份文档解决“怎么答”而不是“答什么”。同样的项目内容，表达方式不对，面试官对你的判断会差很多。

## 一、海思更喜欢什么样的学生

从当前仓库已有材料里抽象，海思/图灵这类高压面试官更喜欢：

1. 基础扎实，不飘。
2. 项目真实，不夸张。
3. 证据清楚，能落到代码、测试、波形和结果。
4. 边界明确，知道什么做了、什么没做。
5. 遇到追问能把问题收敛到模块和信号级，不泛泛而谈。

一句话版本：

**基础扎实、项目真实、边界清楚、能把事情做通的工程型实习生。**

## 二、海思想听到的答题结构

默认结构只有三步：

1. 结论。
2. 证据。
3. 边界。

示例：

“结论先说，我把 SoC 控制链路和验证闭环做通了。证据上，控制在 `src_axi/tpu_frontend_axil.sv` 和 `src_axi/tpu_soc.sv`，e2e 在 `test/test_tpu_soc_axil_e2e.py`，结果是 `41/41 PASS`。边界上，这套验证当前最扎实的是 Q8.8、2x2、2-layer MLP/XOR 这条链路。”

## 三、海思不喜欢什么样的说法

### 1. 空洞堆术语

错误：

“我对 TPU 架构、编译器、验证方法学、后端都有深入研究。”

更稳：

“我当前做得最扎实的是 SoC 集成、验证闭环和 debug 收敛；更大规模映射和通用 compiler 我知道方向，但不会说成已经做完。”

### 2. 把 prototype 说成产品

错误：

“这个设计已经接近产品级 SoC。”

更稳：

“这是一个最小可编程 TPU SoC 原型，重点是把控制、执行和验证闭环跑通。”

### 3. 一被质疑就防御

错误：

“虽然这是学生项目，但也很复杂……”

更稳：

“规模不大，但工程问题是真实的。我的重点不是证明规模，而是证明我能把系统做通、把 bug 收敛住。”

## 四、海思高压面里最加分的态度

### 1. 承认边界

推荐说法：

“这个点我了解原理，但我当前真正做深的是现有 tiny-tpu 原型的控制、验证和 debug 收敛；再往更泛化的方向我可以讲思路，但不会说成自己已经做过完整实现。”

### 2. 承认早期判断不一定对

推荐说法：

“最后那个 H1 问题我一开始也怀疑是 bias 错位，但我没有停在猜测上，而是继续用 directed bench 证伪，最后定位到 UB weight hold-cycle。”

### 3. 不装会

推荐说法：

“UVM 方法学我知道基本思想，但我当前最扎实的验证工作是 cocotb、reference model、scoreboard 和系统级问题定位。”

## 五、最容易扣分的 8 个点

1. 把 scheduler 讲成完整编译器。
2. 把 `41/41 PASS` 讲成“所有情况 fully verified”。
3. 不知道 `41 项`具体是什么。
4. 不知道 `12 epoch` 为什么不能逐 bit 对齐。
5. 不知道 `vpu_pathway` 为什么要 latch。
6. 不知道 `wait_after` 为什么不能隐式推断。
7. 不知道 pipeline 为什么插在 `VPU -> UB` 边界。
8. 对基础题只会背定义，不会落回项目。

## 六、模拟面试顺序

建议每次模拟都按这个顺序走：

1. 自我介绍，1 分钟。
2. 项目介绍，3 分钟。
3. 项目深挖，10 分钟。
4. 验证与 debug，8 分钟。
5. 综合与 PPA，8 分钟。
6. 基础题与手撕代码，10 分钟。
7. 高压追问，5 分钟。

## 七、你要主动练的 12 句高质量回答

1. “我更偏工程型实习生，习惯先给结论，再补证据和边界。”
2. “这个项目我最想强调的不是规模，而是工程闭环。”
3. “最核心的文件是 `src_axi/tpu_frontend_axil.sv`，因为它把原始核变成了可编程加速器。”
4. “scheduler 是当前原型的 stage-level 方案，我不会把它讲成通用 compiler。”
5. “我不是只验证 H1，而是把 H1、dZ2、dZ1 和 UB 参数都拉进了 e2e。”
6. “多 epoch 用例更看系统层收敛，而不是逐 bit 对齐。”
7. “最后一个 H1 bug 最难的地方在于它一开始很像 bias 问题，但最终根因并不在那里。”
8. “`vpu_pathway` 要 latch，是因为控制脉冲和数据到达不同拍。”
9. “`wait_after` 要显式编码，因为等待语义是调度层决策，不应该让状态机靠猜。”
10. “我保留原始 `tpu.sv` 作为基线，再单独做 pipeline 变体，是为了 A/B 对照和可回滚。”
11. “这个设计现在最不工程化的地方，我会优先讲 scheduler 泛化、AXI-Lite 数据面限制，以及覆盖率和断言还没闭合。”
12. “我希望面试官最后觉得我是那种能把问题收敛到模块和信号级的人。”

## 八、模拟面试自检表

每次模拟后，只看下面 10 个问题：

1. 我有没有先说结论？
2. 我有没有给文件和测试名？
3. 我有没有说具体数字？
4. 我有没有主动说边界？
5. 我有没有把猜测和根因混在一起？
6. 我有没有说成“大而全”？
7. 我有没有在不会的点上硬撑？
8. 我有没有把基础题和项目联系起来？
9. 我有没有在 bug 故事里说清“现象 -> 根因 -> 修复 -> 验证”？
10. 我最后给人的感觉是工程型还是包装型？

## 九、最后的临场提醒

- 说慢一点，比说满更重要。
- 先把主线说顺，再进入细节。
- 被打断不可怕，说明对方开始跟你的主线了。
- 不会的点直接收口，不要防御。
- 海思想听到的是工程判断、证据意识和边界意识。
