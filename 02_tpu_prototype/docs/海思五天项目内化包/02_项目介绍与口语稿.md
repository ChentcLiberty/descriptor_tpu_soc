# 项目介绍与口语稿

这份文档只解决一个问题：项目开口怎么讲。海思风格下面，第一分钟比后面十分钟更重要，因为面试官会先判断你是不是“工程型、证据型、边界清楚”的学生。

## 一、统一人设

你对外统一人设只有一句：

**我是偏工程型的实习生，基础不飘，项目真实，习惯先讲结论、再给证据、最后说明边界。**

## 二、项目一句话结论

**我把一个原来主要靠 testbench 直驱的 tiny-tpu RTL，补成了带 AXI-Lite 前端和 IMEM sequencer 的 SoC 原型，并把 forward、backward、参数更新和多 epoch 训练闭环验证跑通了。**

## 三、30 秒版本

“这个项目我做的重点不是单个算子，而是把原来偏 testbench 直驱的 tiny-tpu 核补成 SoC 可控、可验证的原型。我在 `src_axi/tpu_frontend_axil.sv` 和 `src_axi/tpu_soc.sv` 里补了 AXI-Lite 前端、IMEM 和 4 状态 sequencer，再用 `compiler/scheduler.py` 生成 59 条指令驱动当前 2x2、Q8.8 的两层 MLP 流程。验证上我做了两层：单次 e2e 用例 `test/test_tpu_soc_axil_e2e.py` 是 `41/41 PASS`，多 epoch 用例 `test/test_tpu_soc_axil_train_convergence.py` 在 12 个 epoch 后把 loss 从 `0.2529` 降到 `0.1777`，最终 XOR 预测收敛到 `(0,1,1,0)`。”

## 四、90 秒版本

“这个项目最核心的目标，是把原来只能靠 testbench 逐拍驱动的 tiny-tpu RTL，补成能由寄存器配置、写 UB、写 IMEM、启动 sequencer、最后回读结果的 SoC 化原型。我的工作主要分三块。第一块是 SoC 前端和控制链路，我在 `src_axi/tpu_frontend_axil.sv:6-28` 定了寄存器映射和指令格式，在 `src_axi/tpu_frontend_axil.sv:116-210` 实现了 `IDLE / DISPATCH / WAIT / ADVANCE` 这套 sequencer，并在 `src_axi/tpu_soc.sv:86-188` 里把 frontend 和原始 `tpu` 核接起来。第二块是指令化执行，我用 `compiler/scheduler.py:111-259` 生成前向、反向和 update 的阶段级 schedule，再由 `compiler/encode_instrs.py:36-67` 编成 32-bit 指令，最后写到 `compiler/out/imem.hex`。第三块是验证闭环。我在 `test/test_tpu_soc_axil_e2e.py:274-413` 里做了 AXI 写 UB、写 IMEM、启动 sequencer、检查 H1/dZ2/dZ1 和 UB 参数的 e2e 回归，结果是 `41/41 PASS`；又在 `test/test_tpu_soc_axil_train_convergence.py:318-367` 里做了 12 epoch 收敛验证，loss 从 `0.2529` 降到 `0.1777`，最终预测是 `(0,1,1,0)`。所以我更愿意把这个项目定义成‘系统集成 + 验证闭环 + bug 收敛’，而不是只做了一个接口壳。”

## 五、3 分钟版本

“这个项目最开始的问题是，原始 tiny-tpu 更偏模块级或 testbench 直驱验证，能证明基本功能，但距离一个可编程、可启动、可观测的 SoC 形态还差一层。我做的第一步，是补一条 AXI-Lite 控制路径，让外部可以通过寄存器写 UB 数据、写 IMEM、拉起 sequencer。这里最核心的文件是 `src_axi/tpu_frontend_axil.sv` 和 `src_axi/tpu_soc.sv`。前者在 `:6-28` 定义了寄存器和指令格式，在 `:141-210` 实现了 sequencer，对 `wait_after` 和 `vpu_drain` 这类控制语义做了时序化处理；后者在 `:54-188` 负责把 frontend 输出桥接到原始 TPU 核。

第二步，我把当前原型的执行流程指令化了。`compiler/scheduler.py` 在 `:87-105` 先把边界锁死，只支持当前 2x2、2-lane、两层 MLP、training flow；再在 `:124-259` 生成 forward layer1、transition layer2、backward/update 这些命令序列。`compiler/encode_instrs.py` 在 `:36-67` 把这些命令压成 32-bit IMEM 指令。这里我会主动守边界：这是 stage-level scheduler，不是通用编译器。

第三步，也是最有说服力的一步，是验证闭环。我不是只验证一个输出点，而是做了两层验证。`test/test_tpu_soc_axil_e2e.py` 在 `:147-194` 里精确复刻 UB 双 lane 数据装载和 IMEM 写入，在 `:353-413` 里检查 H1、dZ2、dZ1、UB 写回以及参数内容，总共 `41/41 PASS`。`test/test_tpu_soc_axil_train_convergence.py` 在 `:193-202` 支持 repeated-start 跑多个 epoch，在 `:318-367` 里验证训练收敛，12 个 epoch 后 loss 从 `0.2529` 到 `0.1777`，最终 XOR 预测收敛到 `(0,1,1,0)`。

项目里最体现工程能力的部分是 debug。比如 `vpu_pathway` 不保持的问题，本质是控制脉冲只有一拍，但 systolic 输出几拍后才到 VPU，所以我在 `src_axi/tpu_frontend_axil.sv:155-157` 对 `vpu_pathway_reg` 做了 latch；再比如 sequencer 一开始靠隐式规则等 drain，后来在 `src_axi/tpu_frontend_axil.sv:137-139` 明确编码 `wait_after`，在 `:176-205` 区分等待和推进；最后那个 H1 残余 bug，现象很像 bias 错位，但根因最终落在 UB weight hold-cycle 行为，这种问题如果没有 directed bench 和关键地址 spot check 很难收敛。所以我不会把这个项目讲成大而全的 TPU 架构，而会把它讲成一个真实的系统集成、验证闭环和 bug 收敛项目。”

## 六、面试时固定结构

每次回答都尽量按下面的结构：

1. 结论先说。
2. 证据给文件、测试、结果。
3. 边界主动说明。

示例：

“结论先说，我把 SoC 控制链路和验证闭环做通了。证据上，控制链路在 `src_axi/tpu_frontend_axil.sv` 和 `src_axi/tpu_soc.sv`，e2e 测试在 `test/test_tpu_soc_axil_e2e.py`，结果是 `41/41 PASS`。边界上，这套验证最扎实的是 Q8.8、2x2、2-layer MLP/XOR 这条链路。”

## 七、一定要主动守住的边界

不要说：

- 我做了完整 TPU 编译器。
- 我把任意规模训练 fully verified 了。
- 这个设计已经接近产品级 SoC。
- 183.91MHz 就是默认版本的现成结果。

更稳的说法：

- 我做的是当前 tiny-tpu 原型的 schedule 生成和 IMEM 驱动链路。
- 我把当前 2x2、Q8.8、2-layer MLP/XOR 这条闭环做扎实了。
- 我做过一个独立 pipeline 变体，证明了 VPU 到 UB 这条关键路径可以被切短。
- 当前项目最强的是 SoC 集成、验证闭环和 bug 收敛，不是“大而全”。

## 八、结尾句

如果面试官问“你希望我怎么理解这个项目”，最稳的结尾句是：

**这个项目不是在比规模，而是在比工程闭环。我能把结论落到代码、测试和结果，也能把边界讲清楚。**
