# 简历逐条证据映射速查

这份文档只做一件事：把你简历上的每一句话，映射到“代码 + 测试 + 结果 + 边界”。

## 一、项目标题

### 简历表述

`MLP 脉动阵列加速器 RTL 设计、接口封装与验证`

### 你怎么展开

- RTL 设计：`src/pe.sv`、`src/unified_buffer_v3.sv`、`src/tpu.sv`
- 接口封装：`src_axi/tpu_frontend_axil.sv`、`src_axi/tpu_soc.sv`
- 验证：`test/test_tpu_soc_axil_e2e.py`、`test/test_tpu_soc_axil_train_convergence.py`

### 边界

- 这是当前 tiny-tpu 原型，不是大规模量产加速器。

## 二、Bullet 1：微架构与 RTL 实现

### 简历表述

“完成 2×2 Weight Stationary 脉动阵列 TPU 原型 RTL 设计，实现 PE 双缓冲权重寄存器、MAC 数据通路、部分和累加及 valid 级联；完成 Unified Buffer 双 lane 写入、事务式读出与转置供数，打通 forward / backward / update 数据通路。”

### 代码证据

- PE 双缓冲：`src/pe.sv:31-35`、`src/pe.sv:69-79`
- MAC 与 valid：`src/pe.sv:36-48`、`src/pe.sv:83-90`
- TPU top 数据通路：`src/tpu.sv:54-73`、`src/tpu.sv:75-182`
- UB 读写与梯度更新：`src/unified_buffer_v3.sv:19-37`、`src/unified_buffer_v3.sv:148-170`、`src/unified_buffer_v3.sv:212-235`

### 面试可直接说的结论

“Weight Stationary 在我的项目里不是抽象名词，而是 PE 里有 active/inactive 双缓冲权重寄存器，UB 负责喂 input/weight/bias/Y/H 和 update 回写，`tpu.sv` 再把这些子模块接成完整数据通路。”

### 边界

- 当前规模锁在 2x2、Q8.8、2-layer MLP/XOR。

## 三、Bullet 2：SoC 接口封装与控制流

### 简历表述

“设计 AXI-Lite 控制封装层，定义 32-bit IMEM 指令格式，完成 4 状态 sequencer FSM（IDLE / DISPATCH / WAIT / ADVANCE）及 `vpu_drain` 控制；配套编写 Python 调度器与编码脚本，自动生成 59 条指令，覆盖前向、反向传播与参数更新控制序列。”

### 代码证据

- 寄存器映射与指令格式：`src_axi/tpu_frontend_axil.sv:6-28`
- sequencer 状态机：`src_axi/tpu_frontend_axil.sv:116-210`
- `vpu_drain`：`src_axi/tpu_frontend_axil.sv:152-153`
- `wait_after`：`src_axi/tpu_frontend_axil.sv:137-139`
- 顶层桥接：`src_axi/tpu_soc.sv:86-188`
- scheduler：`compiler/scheduler.py:111-259`
- encoder：`compiler/encode_instrs.py:36-67`

### 面试可直接说的结论

“我做的不是单个寄存器 bank，而是把 AXI-Lite、IMEM、sequencer 和 control decode 串成了一套可编程控制链路。”

### 边界

- scheduler 是 stage-level 原型，不是通用编译器。
- AXI-Lite 是控制面，不是高吞吐数据面。

## 四、Bullet 3：验证闭环与系统调试

### 简历表述

“基于 cocotb + NumPy 16-bit Q8.8 参考模型，完成 directed VCS testbench、AXI SoC 端到端验证及多 epoch XOR 训练验证；定位并修复多类系统级问题，最终 AXI e2e 41/41 PASS，并在当前 2-2-1 XOR 场景下实现 12 epoch 收敛，同时补充 repeated-start 压力检查。”

### 代码证据

- reference model：`test/test_tpu_soc_axil_e2e.py:43-104`
- UB/IMEM 装载：`test/test_tpu_soc_axil_e2e.py:147-194`
- e2e scoreboard：`test/test_tpu_soc_axil_e2e.py:379-413`
- 多 epoch：`test/test_tpu_soc_axil_train_convergence.py:13-16`、`test/test_tpu_soc_axil_train_convergence.py:193-202`、`test/test_tpu_soc_axil_train_convergence.py:318-367`
- directed VCS：`test/tb_tpu_h1_bias_fix.sv`

### 面试可直接说的结论

“我不是只验证最终输出，而是把 H1、dZ2、dZ1、UB 参数和多 epoch 收敛都拉进了验证闭环。”

### 结果证据

- `41/41 PASS`
- `12 epoch: loss 0.2529 -> 0.1777`
- 最终预测 `(0,1,1,0)`

### 边界

- 多 epoch 更看系统层收敛，不等于每轮逐 bit 完全一致。

## 五、Bullet 4：前端综合与时序优化

### 简历表述

“基于 SMIC 180nm `tt_1v8_25c` 工艺库完成综合脚本、频点搜索与 SDC 约束，定位 VPU→UB 跨模块关键路径；在 VPU writeback / UB write 接口引入 pipeline 变体完成时序切分，pre-layout Fmax 由 164.10 MHz 提升至 183.91 MHz（+12.07%），QoR 报告 Levels of Logic 由 49 降至 38。”

### 代码证据

- baseline top：`src/tpu.sv:75-182`
- pipeline variant：`src/pipeline_exp/tpu_pipeline.sv:57-210`
- inserted stage：`src/pipeline_exp/vpu_ub_pipe_stage.sv:18-30`
- 综合总结：`docs/SYN_CODE_CHANGES.md`

### 面试可直接说的结论

“我不是一上来就改原始 top，而是保留 baseline，再单独做 pipeline 变体，最后用 timing report 证明关键路径确实被切短。”

### 边界

- 183.91MHz 是 pipeline 变体的实验结果，不是默认 AXI 路径的现成量产结论。

## 六、Bullet 5：面积与功耗优化

### 简历表述

“通过层次化面积报告识别 UB register file 为面积热点，采用 buffer 深度 `128→32` 与 operand isolation 两步优化，总面积由 839K 降至 454K μm²（-46%），前端功耗降低 2.73 mW（-9.4%），频率保持不变。”

### 证据来源

- `career/interview_prep/03_综合时序与PPA优化复盘.md`
- `docs/SYN_INTERNALIZE.md`

### 面试可直接说的结论

“我不是靠感觉说热点，而是通过层次化面积报告定位 UB register file，再做减深度和 operand isolation 两步优化。”

### 边界

- 这块更适合讲结论、思路和 trade-off，不要硬编没有打开过的脚本细节。

## 七、个人总结

### 简历表述

“偏工程型实习生风格，习惯先给结论，再补证据和边界。能够结合代码、波形、参考模型、测试结果与综合报告定位系统问题并完成设计收敛，对未做深的点会明确说明适用范围。”

### 你怎么证明这不是空话

- 先结论：项目介绍用 30 秒 / 90 秒 / 3 分钟三个版本都能做到。
- 再证据：每个 bullet 都能落到代码、测试和结果。
- 再边界：scheduler、AXI-Lite、网络规模、pipeline 变体的边界都能主动说。

## 八、最应该背下来的 5 句“证据型表达”

1. “SoC 前端和控制链路的核心文件是 `src_axi/tpu_frontend_axil.sv`。”
2. “e2e 不是只看 H1，而是把 H1、dZ2、dZ1 和 UB 参数都拉进了 scoreboard。”
3. “多 epoch 用例主要验证系统层收敛，而不是每轮逐 bit 对齐。”
4. “最难 bug 最后落在 UB weight hold-cycle，而不是我一开始猜的 bias 对齐。”
5. “pipeline 变体保留了 baseline，方便 A/B 对照和可回滚。”
