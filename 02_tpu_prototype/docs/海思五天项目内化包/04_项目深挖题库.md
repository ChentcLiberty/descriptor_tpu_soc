# 项目深挖题库

这份题库不是追求“全背”，而是把高频、高风险、最容易被连续追问的问题收束起来。默认答题结构：先结论，再证据，再边界。

## 一、项目定位类

### Q1：你这个项目最核心的工作是什么？

A：核心不是只写一个接口壳，而是把原来偏 testbench 直驱的 tiny-tpu RTL，补成了带 AXI-Lite 前端、IMEM 和 sequencer 的 SoC 原型，并把 forward、backward、参数更新和多 epoch 训练闭环验证跑通了。证据上，控制链路在 `src_axi/tpu_frontend_axil.sv` 和 `src_axi/tpu_soc.sv`，e2e 在 `test/test_tpu_soc_axil_e2e.py`，结果是 `41/41 PASS`；多 epoch 用例在 `test/test_tpu_soc_axil_train_convergence.py`，12 个 epoch 后 loss 从 `0.2529` 降到 `0.1777`。边界上，当前最扎实的是 Q8.8、2x2、2-layer MLP/XOR 这条链路。

### Q2：为什么你的项目标题里有“RTL 设计”？

A：因为我不只是拿现成核做验证，而是实际理解并修改了核心 RTL，还扩展了 SoC 前端。证据上，我能落到 `src/pe.sv`、`src/unified_buffer_v3.sv`、`src/tpu.sv`、`src_axi/tpu_frontend_axil.sv` 这些文件去讲具体信号和时序行为。

### Q3：为什么你的项目标题里有“SoC 集成”？

A：因为我做的不只是把算子跑起来，而是做了寄存器配置、UB 装载、IMEM 装载、sequencer 启动和状态回读这整条控制链路。最能证明这一点的文件是 `src_axi/tpu_frontend_axil.sv` 和 `src_axi/tpu_soc.sv`。

## 二、架构取舍类

### Q4：为什么用 Weight Stationary，不用 Output Stationary？

A：当前原型重点是小规模 MLP 路径验证，Weight Stationary 最适合突出权重 preload、bank 切换和阵列内部流动机制。对 2x2 原型来说，它也更容易暴露权重装载时序和列间错拍问题。

### Q5：为什么是 2x2，不是 8x8？

A：2x2 是当前项目“功能、时序、验证成本”之间的最小可闭环配置。它已经足够暴露真实问题，例如 col1/col2 错拍、VPU pathway 保持、UB 地址/时序以及 wait 语义。如果一开始上更大规模，很容易把问题复杂度堆高，但闭环做不扎实。

### Q6：为什么选 AXI-Lite，不选 AXI4 或 AXI4-Stream？

A：因为当前第一目标是把原型做成可编程、可启动、可观测的 SoC 组件，而不是追求高吞吐数据搬运。AXI-Lite 足以承载寄存器、IMEM 和小规模 UB 数据写入，复杂度也更适合作为 bring-up 第一版接口。

### Q7：为什么要 IMEM 指令流，而不是每一步都寄存器驱动？

A：如果完全靠寄存器逐拍控制，软件参与会过重，复杂流程很难自动化。IMEM + sequencer 把流程控制下沉到硬件，使 forward/backward/update 可以作为一个连续 schedule 运行，也方便用 Python scheduler 自动生成指令。

## 三、SoC 控制链路类

### Q8：4 状态 sequencer FSM 为什么是必要的？

A：因为 sequencer 不只是 PC++。有些指令发出去后必须等一段输出流结束，所以要有 `WAIT` 态；无须等待的指令则直接 `ADVANCE`。因此至少要区分 `IDLE / DISPATCH / WAIT / ADVANCE`。

### Q9：`vpu_drain` 是什么？

A：它表示一段 VPU 有效输出流结束，对应 `vpu_valid_prev && !tpu_vpu_valid_in`。在 `src_axi/tpu_frontend_axil.sv:152-153` 里实现。它的作用是告诉 sequencer 某条需要等待的指令已经真正结束了。

### Q10：为什么 `wait_after` 要显式编码？

A：因为等待语义本质上是调度层决策，不应该让硬件状态机靠猜。项目里已经真实遇到过 drain 被错误消费导致卡死的问题，后来在 `src_axi/tpu_frontend_axil.sv:137-139` 明确用 bit[23] 编码，控制语义才稳定下来。

### Q11：为什么 `vpu_pathway` 要 latch？

A：因为 UB_RD 指令只 dispatch 一拍，但 systolic 结果要过几拍才到 VPU。如果不在 `src_axi/tpu_frontend_axil.sv:155-157` 把 pathway 锁存住，控制信号会先消失，数据真正到来时 VPU 路径已经回到 0，功能必然不对。

## 四、scheduler / IMEM 类

### Q12：scheduler 是不是你做的编译器？

A：不是完整通用编译器。我做的是当前 tiny-tpu 原型的 stage-level scheduler 和 IMEM 编码链路。`compiler/scheduler.py:87-105` 已经把边界锁死，只支持 2x2、2-lane、两层 MLP、training flow。

### Q13：59 条指令是怎么来的？

A：先由 `scheduler.py` 生成命令序列，再由 `encode_instrs.py` 压成 32-bit 指令，最终输出到 `compiler/out/imem.hex`。不是手写 magic number，而是“规格 -> schedule -> bitfield -> IMEM”这条链路生成出来的。

### Q14：32-bit 指令都编码了什么？

A：核心字段包括 opcode、UB 地址、row/col size、transpose、ptr_sel、vpu_pathway 和 `wait_after`。对应 `compiler/encode_instrs.py:48-67` 和 `src_axi/control_unit.sv:75-83`。

### Q15：为什么 wait 命令不编码进 IMEM？

A：当前实现里 wait 作为调度语义存在，不是单独的一条算术/控制 opcode。`encode_instrs.py:41-42` 直接返回 `None`，由 sequencer 基于 `wait_after` 和 `vpu_drain` 来处理等待逻辑。

## 五、验证真实性类

### Q16：`41/41 PASS` 具体是哪 41 项？

A：包括 H1 的 8 项流式结果、dZ2 的 4 项、dZ1 的 8 项、UB 中 dZ2 和 dZ1 的写回检查，以及 UB 中 W1/B1/W2/B2 的参数检查。关键检查逻辑在 `test/test_tpu_soc_axil_e2e.py:381-413`。

### Q17：你怎么证明不是 cocotb 自己算错了？

A：我做了三层约束。第一，Q8.8 reference model 独立生成结果；第二，关键 UB 地址做 spot check；第三，最后一个顽固问题用 directed VCS bench 交叉验证。所以不是只依赖一份 Python 结果。

### Q18：为什么 e2e 不是完全 bit-exact？

A：流式数值路径按 fixed-point 容差比较更适合验证整体数据通路是否对齐，而 UB 中的参数和关键 word 我是精确检查的。这比单纯追求所有流都逐 bit 一致更符合 fixed-point 多级流水线的真实误差模型。

### Q19：为什么多 epoch 不要求每一轮参数都和参考模型完全一致？

A：因为 fixed-point 截断、tile 更新顺序和数据流次序差异会在多轮训练中逐步累积。对多 epoch 用例，更关键的是 loss 是否下降、最终分类是否正确、更新方向是否合理，而不是每一轮每个参数都逐 bit 对齐。

### Q20：你怎么证明训练结果可信？

A：我把训练验证拆成两层。单次 e2e 用例强对齐 H1、dZ2、dZ1 和 UB 参数，证明一次闭环是对的；多 epoch 用例看系统层 loss 和最终分类是否收敛，证明这条链路不只是“能跑一遍 forward”，而是真的能形成训练闭环。

## 六、update 和边界类

### Q21：你说做了 update，怎么证明参数真的在变？

A：可以直接回到测试证据。`test/test_tpu_soc_axil_e2e.py:404-411` 会检查 UB 中更新后的 `W1/B1/W2/B2`；`test/test_tpu_soc_axil_train_convergence.py:318-367` 会在每个 epoch 后读回 UB 参数，再计算新的 loss 和预测。所以更稳的说法不是“我感觉 update 通了”，而是“我有单次 e2e 的参数检查，也有多 epoch 的收敛行为证据”。边界上，我不会扩成“任意学习率、任意网络都 fully verified”。

### Q22：这个项目最大的边界是什么？

A：三个边界最重要。第一，网络规模锁在 2x2、2-layer MLP/XOR。第二，scheduler 是原型级 stage-level scheduler，不是通用 compiler。第三，AXI-Lite 是 bring-up 控制面，不是高吞吐数据面。

### Q23：如果让你继续往下做，你下一步做什么？

A：我会优先做三件事：补不同 learning rate 和不同初始化下的 update 回归；把 scheduler 从固定规模扩到更一般的 tile 映射；补 `reset in-flight`、AXI back-to-back transaction 和覆盖率收敛。

## 七、debug 类

### Q24：你项目里最难的 bug 是什么？

A：最后一个 H1 残余 bug。它一开始现象上很像 bias 错位，因为只错第二列两个点；但继续做 directed VCS bench、对齐 host write 顺序、spot check UB 地址、观察最后一拍 weight valid 后，最终定位到是 UB weight hold-cycle 多保留了一拍 valid/data，导致 PE22 的 shadow weight 被覆盖。

### Q25：为什么一开始会判断成 bias 问题？

A：因为从表面现象看，第二列偏差和列间错拍非常像 bias 对齐问题，这个假设并不荒唐。但它只是“解释得通”的假设，不是最终根因。这个故事最能体现的是我后来没有停在猜测上，而是继续搭 directed case 去证伪。

### Q26：host write 无效这个问题你怎么定位的？

A：现象是 cocotb 已经写了 UB data/push，但 `wr_ptr` 不动、UB memory 仍是 0。后面我把问题收窄到 frontend 到 TPU 的 host valid 没被 UB 采到，最终在 `src_axi/tpu_soc.sv:57-66` 用 scalar bridge 做了桥接，问题才收敛。

### Q27：`vpu_pathway` 不保持的问题你怎么解释？

A：本质上是控制和数据不同拍。指令 dispatch 当拍 pathway 正确，但 systolic 数据几拍后才到 VPU。如果不 latch，后面几拍数据会按 pathway=0 走错路。这个 bug 很适合用来回答“为什么你说自己能把问题收敛到信号级”。

### Q28：sequencer 为什么会卡死？

A：一开始等待语义写得太隐式，前一条指令把 drain 提前消费掉，后一条还在等，状态机就永远出不来。后来改成显式 `wait_after`，并把等待条件明确绑定到 `vpu_drain` 上，状态机才稳定。

## 八、综合 / PPA 类

### Q29：你怎么找到关键路径的？

A：先看 timing report，确认最差路径跨哪些模块；然后发现瓶颈集中在 VPU 到 UB 写回这条链路；最后决定在模块边界插寄存器，而不是深入改 VPU 或 UB 内部逻辑。

### Q30：为什么 pipeline 插在 VPU 和 UB 之间？

A：因为 timing report 证明瓶颈正好跨这两者，把寄存器切在模块边界能有效缩短组合深度，同时又不需要重写内部逻辑，工程上最可控。

### Q31：为什么不直接改原始 `tpu.sv`？

A：为了保留功能基线，便于 A/B 对照。原始 `src/tpu.sv` 保持 baseline，`src/pipeline_exp/tpu_pipeline.sv` 和 `vpu_ub_pipe_stage.sv` 做 timing experiment，这样更工程化，也更容易验证和回滚。

### Q32：你怎么讲面积和功耗优化？

A：先说结论：热点在 UB register file；再说动作：buffer 深度 `128 -> 32` 和 operand isolation；最后给结果：面积从 `839K` 降到 `454K μm²`，动态功耗下降 `2.73mW`。不要乱编脚本细节，把重点放在“怎么找热点、为什么这么做”。

## 九、扩展性类

### Q33：如果扩到 8x8，最先爆什么？

A：我会先看四件事：UB 读写带宽是否还匹配；weight load 和 switch 时序能不能继续靠固定 NOP；VPU 到 UB 写回是否需要更系统的握手；scheduler 是否还能靠现在的规则维持可维护性。

### Q34：如果做产品级 SoC，你先补什么？

A：先补更强的数据搬运接口；再补断言、覆盖率和更完整的 corner/模式综合；同时把 scheduler 向更可扩展的 compiler 前端演进，并增加更多 learning rate / 初始化组合下的训练回归。

## 十、态度类

### Q35：如果面试官说“这只是学生项目”，你怎么接？

A：我不会防御。我会强调这个项目规模不大，但工程问题是真实的：SoC 封装、指令链路、系统级 debug、跨仿真器验证、关键路径定位和 pipeline 变体都是真实问题。对实习生来说，我更想证明的是能不能把事情做通、把问题收敛住。

### Q36：如果面试官指出你设计还不够工程化呢？

A：直接承认边界，例如 scheduler 还不够泛化、AXI-Lite 不适合高吞吐数据搬运、覆盖率还没闭合。然后补一句：当前阶段的重点是先把最小可编程 TPU SoC 闭环跑通，并把关键 bug 收敛到位。

### Q37：你希望面试官最后怎么评价你？

A：我希望对方觉得我是那种基础不飘、项目真实、能把问题收敛到模块和信号级，而且边界很清楚的人。对实习岗来说，这比把单点包装得很强更重要。
