# 项目详解与代码定位

这份文档解决两个问题：

1. 项目到底由哪些模块构成。
2. 每句项目描述能落到哪段代码。

## 一、先建立全局结构

整个项目可以拆成 5 块：

1. 计算核心：`src/`
2. SoC 前端：`src_axi/`
3. 指令生成：`compiler/`
4. 验证：`test/`
5. 结果与总结：`docs/`

对应你面试里的说法：

“我不是只改了一个模块，而是把 compute、control、schedule、verification 这四层串起来了。”

## 二、项目主数据流怎么走

### 1. Host 把数据写进 UB

关键代码：

- `src_axi/tpu_frontend_axil.sv:12-20`
- `src_axi/tpu_frontend_axil.sv:66-71`
- `src_axi/tpu_soc.sv:57-66`
- `src/tpu.sv:75-126`

你要讲的点：

- AXI-Lite 寄存器里有 `UB_DATA`、`UB_PUSH`、`IMEM_ADDR`、`IMEM_W0`、`IMEM_WE`。
- frontend 把 host write 拆成 lane0/lane1 两路标量信号。
- `tpu_soc.sv` 再把这两路标量桥接成 `tpu.sv` 需要的 unpacked array 输入。
- `tpu.sv` 最后把 host write 真正送进 UB。

### 2. Sequencer 从 IMEM 发控制指令

关键代码：

- `src_axi/tpu_frontend_axil.sv:107-140`
- `src_axi/tpu_frontend_axil.sv:141-210`

你要讲的点：

- `imem`、`pc`、`seq_state`、`seq_instr_pulse` 都在 frontend 里。
- `step_pulse` 支持单步，`start_pulse` 支持从 `imem[0]` 自动跑。
- `seq_needs_wait = seq_instr[23]` 明确把等待语义做成了指令级 bit，而不是写死在状态机里。
- `vpu_drain = vpu_valid_prev && !tpu_vpu_valid_in` 用来判断一段输出流结束。

### 3. Control unit 把 32-bit 指令译码成控制信号

关键代码：

- `src_axi/control_unit.sv:6-21`
- `src_axi/control_unit.sv:42-94`

你要讲的点：

- opcode、addr、row、col、transpose、ptr_sel、vpu_pathway 都在这里拆开。
- `UB_RD` 会拉起 `ub_rd_start_in`，同时给出地址、维度、数据来源和 VPU 路径。
- `SWITCH` 只做权重 bank 切换。

### 4. TPU top 把 UB、systolic、VPU 串起来

关键代码：

- `src/tpu.sv:54-73`
- `src/tpu.sv:75-126`
- `src/tpu.sv:128-153`
- `src/tpu.sv:155-182`

你要讲的点：

- `tpu.sv` 是原始 compute top。
- UB 负责存取 X/W/B/Y/H 和写回更新后的参数。
- systolic 负责乘加和数据波前推进。
- VPU 负责 bias、leaky_relu、loss derivative、leaky_relu derivative。

## 三、每个核心模块到底做什么

## 1. `src/pe.sv`

关键位置：

- `src/pe.sv:31-35`
- `src/pe.sv:50-93`

你要会说：

- `weight_reg_active` 和 `weight_reg_inactive` 是双缓冲权重寄存器。
- `pe_accept_w_in` 负责 preload 到 inactive bank。
- `pe_switch_in` 把 inactive 复制到 active，实现 Weight Stationary 的无缝切换。
- `pe_valid_out <= pe_valid_in` 说明 valid 会沿阵列传递，不是只传数据不传时序语义。

高频回答：

“Weight Stationary 在我这里不是概念，而是 PE 里有 active/inactive 双缓冲权重寄存器，先 preload，再 switch。”

## 2. `src/unified_buffer_v3.sv`

关键位置：

- `src/unified_buffer_v3.sv:19-37`
- `src/unified_buffer_v3.sv:93-170`
- `src/unified_buffer_v3.sv:212-235`
- `src/unified_buffer_v3.sv:239-260`

你要会说：

- UB 不只是存数据，还承担 host write、input read、weight read、bias/Y/H read、gradient update 这些职责。
- `wr_ptr` 和各种 `rd_*_ptr`/`time_counter` 决定了什么时候从哪个区域读写。
- `grad_descent_valid_in` 的生成逻辑说明 update 路径并不是拍脑袋接上的，而是和读写计数器严格对齐的。
- 很多系统级 bug 最后都落在 UB 地址和时序上。

高频回答：

“这个项目里 UB 是系统复杂度最高的模块之一，因为它同时承担数据装载、事务式读出、VPU 辅助输入和 update 回写。”

## 3. `src/vpu.sv`

关键位置：

- `src/vpu.sv:4-17`
- `src/vpu.sv:23-45`
- `src/vpu.sv:187-240`

你要会说：

- `vpu_data_pathway` 是 4 bit 控制，不同组合对应不同后处理路径。
- `1100` 对应第一层 forward：bias + leaky_relu。
- `1111` 对应 transition/loss 路径：bias + leaky_relu + loss derivative + leaky_relu derivative。
- `0001` 对应 backward 路径：只经过 leaky_relu derivative。

高频回答：

“VPU 不是一个单模块，而是一条组合式后处理链；面试时我会把它讲成 pathway-controlled post-processing pipeline。”

## 4. `src_axi/tpu_frontend_axil.sv`

关键位置：

- `src_axi/tpu_frontend_axil.sv:6-28`
- `src_axi/tpu_frontend_axil.sv:94-106`
- `src_axi/tpu_frontend_axil.sv:116-210`
- `src_axi/tpu_frontend_axil.sv:212-260`

你要会说：

- 这里同时做了寄存器 bank、IMEM、step/start 控制和 sequencer。
- `vpu_pathway_reg` 是为了解决控制只来一拍但数据几拍后才到的问题。
- `busy_reg`、`seq_running`、`pc`、`seq_state` 是状态机的主证据。

高频回答：

“如果只让我选一个最能证明 SoC 集成能力的文件，我会选 `tpu_frontend_axil.sv`。”

## 5. `src_axi/tpu_soc.sv`

关键位置：

- `src_axi/tpu_soc.sv:54-78`
- `src_axi/tpu_soc.sv:86-139`
- `src_axi/tpu_soc.sv:146-188`

你要会说：

- 它负责把 frontend 和原始 `tpu` 核接起来。
- host write scalar bridge 就在这里，是修复 host write 无效问题的关键一环。
- `learning_rate` 也是在这里真正接到 `tpu` 核上的。

高频回答：

“`tpu_soc.sv` 的价值不是算法，而是桥接 AXI 世界和原始 compute world。”

## 6. `compiler/scheduler.py`

关键位置：

- `compiler/scheduler.py:87-105`
- `compiler/scheduler.py:124-177`
- `compiler/scheduler.py:179-259`

你要会说：

- 一开始就把边界锁死：只支持 2x2、2-lane、两层 MLP、training flow。
- forward layer1、transition layer2、backward/update 都是由这里生成 stage-level command。
- `wait_after` 也在 schedule 层决定，而不是由硬件猜。

高频回答：

“scheduler 的价值不是泛化，而是把当前原型从手工驱动推进到自动生成 schedule 并进硬件执行。”

## 7. `compiler/encode_instrs.py`

关键位置：

- `compiler/encode_instrs.py:4-24`
- `compiler/encode_instrs.py:36-67`
- `compiler/encode_instrs.py:83-103`

你要会说：

- 32-bit 指令的 bitfield 在这里编码。
- `wait` 类型命令不编码进 IMEM，而是由 sequencer 语义处理。
- 最终会生成 `imem.hex` 和 `imem.txt`，不是只停在抽象 JSON。

## 8. `test/test_tpu_soc_axil_e2e.py`

关键位置：

- `test/test_tpu_soc_axil_e2e.py:43-104`
- `test/test_tpu_soc_axil_e2e.py:147-194`
- `test/test_tpu_soc_axil_e2e.py:325-413`

你要会说：

- `reference_forward()` 和 `reference_backward()` 给出 Q8.8 参考路径。
- `load_all_data_axil()` 复刻了 UB 双 lane 装载时序。
- `imem_load()` 负责把 `imem.hex` 真正写进 DUT。
- 最后不仅检查 H1/dZ2/dZ1，还检查 UB 中 dZ2、dZ1 和更新后的参数。

## 9. `test/test_tpu_soc_axil_train_convergence.py`

关键位置：

- `test/test_tpu_soc_axil_train_convergence.py:13-16`
- `test/test_tpu_soc_axil_train_convergence.py:193-202`
- `test/test_tpu_soc_axil_train_convergence.py:318-367`

你要会说：

- `TRAIN_EPOCHS = 12`，目标不是逐 bit 对齐，而是 loss 下降和最终分类正确。
- `run_one_epoch()` 复用 repeated-start 触发多轮训练。
- 每轮读回 UB 中的 W1/B1/W2/B2，再在 Python 里计算新的 loss/pred。

## 四、你必须能脱口而出的 3 条代码证据链

### 证据链 1：SoC 控制链路

- 入口：`src_axi/tpu_frontend_axil.sv:6-28`
- 状态机：`src_axi/tpu_frontend_axil.sv:141-210`
- 顶层桥接：`src_axi/tpu_soc.sv:86-188`
- 验证落点：`test/test_tpu_soc_axil_e2e.py:325-413`

一句话：

“寄存器、IMEM、sequencer 和顶层桥接都在 SoC 前端链路里，e2e 已经证明这条控制链路能真实驱动硬件执行。”

### 证据链 2：训练闭环

- schedule 生成：`compiler/scheduler.py:124-259`
- 指令编码：`compiler/encode_instrs.py:36-67`
- 单次 e2e：`test/test_tpu_soc_axil_e2e.py:353-413`
- 多 epoch：`test/test_tpu_soc_axil_train_convergence.py:318-367`

一句话：

“训练闭环不是一句话，而是 schedule、IMEM、单次 e2e 和多 epoch 收敛四层证据串起来的。”

### 证据链 3：时序优化

- baseline top：`src/tpu.sv:75-182`
- pipeline variant：`src/pipeline_exp/tpu_pipeline.sv:57-210`
- pipeline stage：`src/pipeline_exp/vpu_ub_pipe_stage.sv:18-30`

一句话：

“我没有硬改原始 top，而是保留基线，再单独做 pipeline 变体进行 A/B 对照。”

## 五、最容易被追问的 4 个点

1. `learning_rate_in` 的边界要说清。
2. scheduler 不是通用编译器。
3. `41/41 PASS` 不只是看 H1。
4. pipeline 变体不是默认量产版本。
